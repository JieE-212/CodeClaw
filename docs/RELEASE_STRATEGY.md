# CodeClaw Release Strategy

This document records the stage-four productization decision for how CodeClaw should first reach real users.

## Decision

Start with a local Web workspace that can be packaged later as a desktop app.

The near-term release target is:

1. Local Node service.
2. Browser-based workspace at `http://localhost:4173`.
3. One-command startup and one-command health check.
4. Local project memory, local audit logs, and explicit approval for writes and commands.
5. DeepSeek Flash as the default real-model option, with Pro reserved for complex work or review.

Do not start with a WeChat mini program or pure cloud SaaS as the first real-user release.

## Why This Route

CodeClaw's core value depends on local developer capabilities:

- Read local repositories.
- Respect `.gitignore` and skip local state.
- Read selected source and test files.
- Propose full-file patches.
- Write files only after approval.
- Run local verification commands.
- Persist project memory and audit records locally.

These requirements are difficult or impossible in a WeChat mini program without a companion local service. A pure cloud version would require users to upload source code or connect remote repositories, which raises privacy, security, authentication, and cost questions before the product loop is proven.

The local Web route keeps the current architecture useful and testable while leaving room for a desktop shell later.

## Route Comparison

| Route | Fit Now | Strengths | Main Risks | Decision |
| --- | --- | --- | --- | --- |
| Local Web workspace | High | Already works, easiest to test, preserves local file access | Requires user to run local service | Use now |
| Desktop app shell | Medium | Better user experience, can hide local service details | Packaging, auto-update, signing, platform testing | Next packaging step |
| Pure cloud SaaS | Low | Easy account sharing and centralized billing | Source privacy, remote repo auth, command execution impossible locally | Later, after local workflow stabilizes |
| WeChat mini program | Low | Familiar distribution in China | No direct local repo access, no local command execution, needs companion service | Not first release |
| Mobile app | Very low | Consumer-style access | Poor fit for code editing, local repo and command workflows | Not first release |

## Recommended Release Stages

### Stage 4A - Local Web Trial

Goal: Make the current local Web product safe enough for a small group of real users.

Required before sharing:

- `npm.cmd run trial:ready` passes.
- `npm.cmd run health` passes.
- `npm.cmd run check` passes.
- `npm.cmd test` passes.
- `npm.cmd run package:local-trial` creates a clean package folder.
- Demo flow works from the browser.
- A real project can pass read-only preflight.
- Model panel clearly distinguishes Flash and Pro cost risk.
- Users understand that writes and command runs require approval.

Deliverable:

- A zip or folder users can run locally.
- A short setup guide: `docs/START_GUIDE.md`.
- A first-trial host runbook: `docs/FIRST_TRIAL_RUNBOOK.md`.
- A persona trial review: `docs/PERSONA_TRIAL_REVIEW.md`.
- A local package checklist: `docs/LOCAL_TRIAL_PACKAGE.md`.
- A feedback template: `docs/TRIAL_FEEDBACK_TEMPLATE.md`.
- A ready-to-send invite: `docs/TRIAL_INVITE_MESSAGE.md`.
- A package manifest generated by `scripts/prepare-local-trial.js`.
- A readiness report generated by `scripts/trial-readiness.js`.
- A first-run checklist in the browser UI.
- A restart handoff: `docs/HANDOFF_RESTART.md`.

### Stage 4B - Local Launcher

Goal: Reduce setup friction without changing the architecture.

Current launcher assets:

- `start-codeclaw.cmd`
- `docs/START_GUIDE.md`
- `docs/FIRST_TRIAL_RUNBOOK.md`
- `docs/PERSONA_TRIAL_REVIEW.md`
- `docs/LOCAL_TRIAL_PACKAGE.md`
- `docs/TRIAL_FEEDBACK_TEMPLATE.md`
- `docs/TRIAL_INVITE_MESSAGE.md`

Launcher behavior:

- Check Node version before startup.
- Open the browser automatically.
- Show a clear error if port `4173` is busy.
- Keep `npm.cmd run health` as the release validation command.

Deliverable:

- A local trial package for Windows users.

### Stage 4C - Desktop Shell

Goal: Package the local Web workspace as an app-like experience.

Candidate routes:

- Electron, if app packaging and full desktop integration matter.
- Tauri, if smaller binaries matter and the team is ready for Rust/toolchain overhead.
- A lightweight launcher first, if product feedback is still the main goal.

Recommended order:

1. Lightweight launcher.
2. Electron/Tauri only after 3-5 real local trials show the workflow is worth packaging.

### Stage 4D - Multi-user / Cloud Layer

Goal: Support teams and shared billing without losing local safety.

Do this only after the local workflow is stable.

Possible split:

- Local agent: file access, patching, command execution.
- Cloud service: accounts, billing, model proxy, shared configuration, optional team audit summaries.

Keep source code local by default.

## WeChat Mini Program Position

A WeChat mini program can be useful later as a companion surface, not as the core agent.

Possible future uses:

- View task summaries.
- Approve or reject high-level actions.
- Receive notifications.
- Browse audit summaries.

Not suitable as the first core product because it cannot directly perform the local developer workflow.

## Default Model Policy

Use this policy in product copy and defaults:

- Default: DeepSeek V4 Flash.
- Use Pro only for complex tasks, ambiguous architecture decisions, high-risk changes, or review of a Flash proposal.
- Keep mock available for demos, health checks, and offline workflow testing.
- Avoid making Pro the default for multi-user product usage.

## Release Gate

Before any real-user package is shared:

```bash
npm.cmd run trial:ready
```

Or run the checks manually:

```bash
npm.cmd run health
npm.cmd run check
npm.cmd test
npm.cmd run package:local-trial
```

For a real project trial:

```bash
npm.cmd run pilot:real:preflight -- "C:\path\to\repo" "narrow trial goal"
```

Stop before writes if preflight has blockers, context misses the relevant source/test files, or no verification command is available for a project that should have tests.

## Current Next Step

Prepare the local Web trial package path:

1. Run `npm.cmd run trial:ready`.
2. Review `dist/TRIAL_READINESS_REPORT.json`.
3. Run `npm.cmd run trial:simulate` if no external tester is available.
4. Review `docs/PERSONA_TRIAL_REVIEW.md` for remaining first-user risks.
5. Use `docs/FIRST_TRIAL_RUNBOOK.md` to host one external trial.
6. Share `docs/START_GUIDE.md` and `docs/TRIAL_FEEDBACK_TEMPLATE.md` with the tester.
7. Use `docs/TRIAL_INVITE_MESSAGE.md` for the invite.
8. Trial on one real project in read-only mode first.
9. Collect 3-5 feedback reports after the first trial has no launch blocker.
10. Only then consider desktop shell packaging.
